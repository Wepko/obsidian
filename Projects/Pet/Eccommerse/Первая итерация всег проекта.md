## 🏗️ Итоговая архитектура проекта

Мы спроектировали систему, которая объединяет **три ключевых компонента**, работающих согласованно на едином Laravel API с мультитенантностью.

### 1. Единый бэкенд (Laravel API)
- **Ядро системы**: вся бизнес-логика, база данных, аутентификация.
- **Мультитенантность**: данные всех клиентов (тенантов) разделены по `tenant_id` в одной базе данных.
- **Роли**:
  - **Супер-админ** (вы): полный доступ ко всем тенантам, управление клиентами, фрилансерами, биллингом.
  - **Админ** (клиент/менеджер): доступ только к данным своего тенанта.

### 2. Клиентские сайты (YourNextStore)
- **База**: форк [yournextstore/yournextstore](https://github.com/yournextstore/yournextstore).
- **Назначение**: каждый клиент получает отдельный экземпляр (отдельный деплой) с уникальным дизайном.
- **Деплой**: по домену `client1.com`, `client2.com` и т.д.
- **Кастомизация**: фрилансеры меняют дизайн, структуру, но используют единое API.

### 3. Админка для клиентов (Next.js SaaS Starter)
- **База**: форк [nextjs/saas-starter](https://github.com/nextjs/saas-starter).
- **Назначение**: единое приложение для всех клиентов, где они управляют своим магазином (товары, заказы, пользователи).
- **Доступ**: по единому домену `admin.legends.com/login`. Система определяет тенант по субдомену (например, `admin.client1.com`) или параметру.
- **Роли**: внутри этой админки только роль **admin** (клиент) и **super-admin** (вы, видите все тенанты).

### 4. Ваша внутренняя админка (Filament)
- **Назначение**: только для вас. Управление клиентами (тенантами), фрилансерами, задачами на доработку, биллингом.
- **Расположение**: отдельное приложение (например, `platform.legends.com`).

## 🔄 Как всё взаимодействует

1. **Клиент** (владелец магазина) заходит на `admin.legends.com` (или `admin.client1.com`).
2. React-админка определяет его тенант (по субдомену/параметру) и отправляет запросы в Laravel API с заголовком `X-Tenant-Domain`.
3. Laravel API проверяет права пользователя и возвращает данные **только для этого тенанта**.
4. Клиент управляет товарами, заказами, пользователями.
5. **Покупатели** заходят на `client1.com` (сайт на YourNextStore), видят кастомный дизайн, делают заказы.
6. **Вы** через Filament управляете всей платформой: создаете новых клиентов, нанимаете фрилансеров для доработок, следите за биллингом.

## 🔐 Ключевые технические решения

### 1. Определение тенанта в админке клиентов
- **React-приложение** (SaaS Starter) читает субдомен (`admin.client1.com`) или GET-параметр.
- **Laravel API** получает тенант из заголовка `X-Tenant-Domain` и применяет глобальный скоуп.

### 2. Разграничение доступа
- **Супер-админ** (вы) в админке клиентов видит переключатель тенантов и может управлять любым магазином.
- **Админ** (клиент) видит только свой магазин и не имеет доступа к настройкам платформы.
- **Ваша Filament-админка** полностью отделена и имеет доступ ко всем данным.

### 3. Работа с фрилансерами
- Вы создаете задачу в Filament (например, "Сделать дизайн для client1.com").
- Фрилансер получает доступ к репозиторию `storefront-template`, делает кастомный дизайн.
- Вы деплоите готовый сайт на домен клиента (отдельный Vercel-проект или VPS).

## 🎯 Ключевые преимущества этой архитектуры

- ✅ **Масштабируемость**: каждый компонент развивается независимо
- ✅ **Гибкость**: фрилансеры работают только с фронтендом (YourNextStore), не трогая бэкенд
- ✅ **Безопасность**: четкое разделение данных через мультитенантность
- ✅ **Скорость разработки**: используем готовые стартеры вместо написания всего с нуля
- ✅ **Единый API**: все приложения работают с одним источником правды
