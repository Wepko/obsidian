## Полная схема обработки запроса в Laravel (с уровнем исключений)

```
HTTP запрос
    ↓
[1] Маршрут (Route) + Middleware (глобальные, группы)
    ↓
[2] FormRequest (валидация входящих данных + авторизация)
    ↓
[3] Контроллер (получает уже проверенные данные)
    ↓
[4] Сервис (бизнес-логика, транзакции, вызовы других сервисов)
    ↓
[5] Глобальный Exception Handler (перехватывает исключения с любого уровня)
    ↓ (если исключения нет)
[6] Ресурс (Resource / Transformer) — форматирование ответа
    ↓
HTTP ответ
```

**Важно:** Exception Handler — это не отдельный шаг в цепочке, а **всегда активный слой**, который может прервать выполнение на любом этапе (от middleware до сервиса) и сразу сформировать ответ, минуя ресурс.

## Разберём каждый уровень подробно (с учётом Handler)

### 1. Маршрут и Middleware (самый внешний слой)

**Что здесь происходит:**
- Проверка **наличия прав доступа** к маршруту (аутентификация, роль, IP-фильтры)
- **Логирование** запроса
- **CORS** (для API)
- **Rate limiting** (ограничение частоты запросов)
- **Начало сессии** (для web)
- **Проверка подписи** (для signed URL)
- **Перенаправление** если условие не выполнено (например, не аутентифицирован → на логин)

**Что НЕ делаем на этом уровне:**
- Валидацию полей формы (это позже)
- Бизнес-логику

**Если возникает исключение** (например, `AuthenticationException`), оно передаётся в Handler → Handler возвращает 401 JSON или редирект на логин.

---

### 2. FormRequest (валидация и авторизация)

**Что здесь происходит:**
- **Авторизация** (может ли пользователь выполнить это действие)
- **Валидация** входных данных (правила: required, email, min, max, exists, unique...)
- **Подготовка данных** (например, приведение типов, фильтрация)

**Что НЕ делаем:**
- Бизнес-логику (например, проверку остатков на складе) — это в сервисе
- Работу с БД (кроме простых exists, unique)

**Исключения:** `ValidationException` (ошибки валидации) → Handler преобразует в JSON 422 со списком ошибок.

---

### 3. Контроллер (оркестрация)

**Что здесь происходит:**
- Получение **уже проверенных данных** из FormRequest
- Вызов **одного или нескольких сервисов**
- **Обработка исключений** (необязательно, можно довериться Handler)
- Возврат **ресурса** или редиректа

**Контроллер должен быть ТОНКИМ** (5-10 строк).  
**Рекомендация:** Не использовать try-catch в контроллере, если можно обработать исключения глобально.

```php
class OrderController extends Controller
{
    public function store(StoreOrderRequest $request, OrderService $orderService)
    {
        $order = $orderService->createFromData($request->validated());
        return new OrderResource($order);
    }
}
```

---

### 4. Сервис (бизнес-логика)

**Что здесь происходит:**
- Все **бизнес-правила**, которые требуют состояния БД или внешних систем
- **Транзакции БД**
- **Проверка остатков**, прав на скидку, применение промокодов
- **Вызов других сервисов** (платежи, уведомления, инвентарь)
- **Резервирование**, списание, начисление бонусов
- **Генерация событий** (event)

**Что НЕ делаем:**
- Валидацию входящих данных (она уже в FormRequest)
- Форматирование ответа (это ресурс)
- Прямой доступ к `$request`

**Исключения:** бизнес-исключения (`OutOfStockException`, `PaymentFailedException` и т.д.) → Handler преобразует их в понятные ответы.

---

### 5. Глобальный Exception Handler (App\Exceptions\Handler)

**Что здесь происходит:**
- Перехватывает **любые исключения** на любом уровне (middleware, form request, controller, service)
- **Логирование** ошибки (в файл, sentry и т.д.)
- **Преобразование исключения в HTTP-ответ** (JSON для API, HTML-страница для web)
- **Динамическая смена статуса ответа** (401, 403, 404, 422, 500 и др.)

**Пример реализации для API:**

```php
// app/Exceptions/Handler.php

public function render($request, Throwable $e)
{
    if ($request->expectsJson()) {
        // Ошибки валидации из FormRequest
        if ($e instanceof ValidationException) {
            return response()->json([
                'message' => 'Ошибка валидации',
                'errors' => $e->errors(),
            ], 422);
        }
        
        // Бизнес-исключения из сервиса
        if ($e instanceof OutOfStockException) {
            return response()->json([
                'error' => 'Товар закончился',
                'product_id' => $e->getProductId(),
            ], 422);
        }
        
        if ($e instanceof PaymentFailedException) {
            return response()->json(['error' => $e->getMessage()], 400);
        }
        
        // Ошибки аутентификации из middleware
        if ($e instanceof AuthenticationException) {
            return response()->json(['error' => 'Не авторизован'], 401);
        }
        
        // Все остальные (в т.ч. 404 ModelNotFoundException)
        return response()->json(['error' => $e->getMessage()], 500);
    }
    
    return parent::render($request, $e);
}
```

**Что НЕ делаем:**
- Бизнес-логику (Handler не должен вызывать сервисы)
- Валидацию

---

### 6. Ресурс (форматирование успешного ответа)

**Что здесь происходит:**
- **Преобразование модели/коллекции** в массив/JSON для API
- **Скрытие полей** (пароль, токены)
- **Добавление вычисляемых полей** (например, `formatted_total`, `can_cancel`)
- **Вложение связанных ресурсов** (товары заказа, пользователь)
- **Локализация** (перевод статусов)

**Что НЕ делаем:**
- Бизнес-логику (не вызываем сервисы)
- Валидацию
- Доступ к БД (всё должно быть уже загружено через `with` в контроллере)

**Ресурс вызывается ТОЛЬКО если не было исключения** — то есть только для успешных ответов (200, 201).

## Итоговая таблица: какие проверки на каком уровне

| Уровень | Что проверяет/делает | Примеры | Какие исключения генерирует |
|---------|----------------------|----------|----------------------------|
| **Middleware** | Аутентификация, права доступа к маршруту, лимиты запросов, CORS | `auth`, `role:admin`, `throttle:60` | `AuthenticationException`, `AuthorizationException`, `ThrottleRequestsException` |
| **FormRequest** | Валидация полей (типы, форматы, наличие в БД), авторизация действия | `email\|required`, `exists:products,id` | `ValidationException` |
| **Контроллер** | Оркестрация, вызов сервисов (обычно без try-catch) | `$orderService->create()` | Любые, но обычно пробрасывает дальше |
| **Сервис** | Бизнес-правила (остатки, скидки, транзакции), работа с БД, внешние API | `if ($stock < $quantity) throw` | Кастомные бизнес-исключения |
| **Exception Handler** | Перехват всех исключений → формирование HTTP-ответа (статус, сообщение, структура) | `render()` | (не генерирует, а обрабатывает) |
| **Ресурс** | Форматирование успешного ответа (модель → JSON) | `'total' => $this->total` | (не генерирует) |

## Почему эта архитектура правильная?

1. **Единое место для ошибок** — Handler отвечает за все исключения, не нужно дублировать try-catch.
2. **Чистые контроллеры** — они не засорены обработкой ошибок.
3. **Сервисы могут свободно выбрасывать исключения** — не думая о формате ответа.
4. **Ресурсы работают только с успешными данными** — не нужно проверять наличие ошибок.
5. **Гибкость** — легко изменить формат ошибок для API или добавить новый тип исключения.

## Ваше тестовое задание

В тестовом задании обязательно покажите понимание этой цепочки. Даже если вы не используете Handler явно (Laravel по умолчанию уже имеет базовый), вы можете продемонстрировать, что знаете, как его расширить.

**Для создания заказа** вы будете:
- Валидировать данные в `StoreOrderRequest`
- Вызывать `OrderService->createFromData()` в контроллере
- В сервисе выбрасывать `OutOfStockException` при нехватке товара
- В Handler преобразовывать это исключение в JSON 422
- При успехе возвращать `OrderResource`

Теперь у вас полное представление о всех слоях Laravel, включая обработку ошибок. Это именно то, что ждут от Middleware-фреймворка.
