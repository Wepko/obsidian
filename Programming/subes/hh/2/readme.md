## Для начала: какая у нас проблема?

Представьте колл-центр, где операторы принимают звонки. Это **дорого**:
- Оператору нужно платить зарплату
- Он может работать только 8 часов
- Ему нужно платить сверхурочные
- Он может заболеть или уволиться

**PathVoice** заменяет оператора **искусственным интеллектом**. Но AI не умеет просто так взять и ответить на звонок — ему нужно помочь:

1. Принять звонок из телефонии (Twilio, SIP и т.д.)
2. Распознать речь (ASR — Speech-to-Text)
3. Понять смысл (NLP — Natural Language Processing)
4. Принять решение (что делать: ответить, перевести на оператора, записать голосовую почту)
5. Синтезировать речь (TTS — Text-to-Speech)
6. Отправить ответ обратно в телефонию

Вот тут и появляется наш **оркестратор**.

## Что делает наш PHP-проект?

Представьте оркестр. Есть дирижер (наш PHP-сервис) и много музыкантов (AI-сервисы на Python, базы данных, очереди, телефония).

**Дирижер НЕ играет на инструментах.** Он только машет палочкой:
- Когда вступить скрипкам
- Когда играть громче
- Когда затихнуть

Так и наш PHP-сервис:
- **НЕ распознает речь** (это делает Python-сервис ASR)
- **НЕ понимает смысл** (это делает Python-сервис NLP)
- **НЕ синтезирует голос** (это делает Python-сервис TTS)

**PHP-сервис ТОЛЬКО координирует:**

```
Звонок → PHP получил → "Эй, Python, распознай речь" → 
Python вернул текст → PHP говорит: "Python, пойми смысл" → 
Python вернул интент → PHP решает: "Надо перевести на оператора" → 
PHP говорит телефонии: "Переведи звонок на оператора №5"
```

## А теперь про сложные штуки

### Зачем нужен **gRPC**?

Обычно сервисы общаются по HTTP/REST. Это как отправлять письма:
- Написал письмо
- Положил в конверт
- Отправил почтой
- Ждешь ответ

gRPC — это как **телефонный разговор**:
- Быстрее
- Можно передавать поток данных (например, голос в реальном времени)
- Строгий контракт (как в контракте написано, так и будет)

В нашем проекте gRPC используется для:
- Общения с AI-сервисами (быстро и эффективно)
- Потоковой передачи аудио (клиент говорит, а мы сразу отправляем в распознавание)

### Зачем **RoadRunner + Laravel Octane**?

Обычный PHP работает так:
1. Пришел запрос
2. PHP загрузил фреймворк
3. Фреймворк обработал запрос
4. Все выгрузилось из памяти
5. Конец

Это как каждый раз собирать и разбирать завод заново, чтобы сделать одну деталь.

**Octane** держит приложение в памяти постоянно:
1. Загрузили фреймворк один раз
2. Он висит в памяти
3. Запросы приходят и обрабатываются мгновенно
4. Не надо перезагружать всё каждый раз

**RoadRunner** — это вообще супер-быстрый сервер на Go, который умеет:
- Держать PHP в памяти
- Балансировать нагрузку
- Работать с WebSocket
- И да, он умеет запускать Laravel Octane внутри себя

В итоге: вместо 100 запросов в секунду получаем 1000+ запросов.

### Зачем **RabbitMQ** (очереди)?

Представьте, что одновременно звонят 1000 человек. Наш PHP не может обработать всё сразу — упадет.

Очереди работают как **буфер**:
1. Звонок приходит → кладем в очередь
2. Воркеры забирают по одному звонку
3. Обрабатывают не спеша
4. Если что-то упало — можно обработать снова

Плюс **отложенные задачи**:
- Клиент не взял трубку → положим в очередь задачу "перезвонить через 2 часа"
- Через 2 часа задача автоматически выполнится

### Зачем **OpenSearch** (логи)?

Когда у вас 1000 звонков в минуту, найти один конкретный разговор — это как иголку в стоге сена.

OpenSearch (тот же Elasticsearch) позволяет:
- Искать по любому полю (номер телефона, дата, результат)
- Строить графики (сколько звонков, какие успешные)
- Хранить терабайты логов

### Зачем **Datadog** (метрики)?

Нужно понимать, как работает система:
- Сколько звонков в секунду
- Не упал ли AI-сервис
- Не тормозит ли база
- Сколько времени обрабатывается звонок

Datadog собирает все метрики в одном месте и показывает дашборды:
```
Звонки: 150/сек
Среднее время ответа AI: 230ms
Ошибок: 0.01%
Свободно памяти: 4.2GB
```

## Как это работает по шагам (user story)

**Шаг 1. Звонок от клиента**
Клиент набирает номер. Телефония (Twilio) шлет HTTP-запрос в наш PHP:
"Эй, у меня звонок с номера +1234567890, вот что он говорит..."

**Шаг 2. PHP принимает запрос**
Наш RoadRunner мгновенно будит PHP (он уже в памяти) и передает запрос.

**Шаг 3. PHP ищет клиента**
PHP лезет в базу данных: "Был ли у нас этот номер раньше?"
- Если да — достаем историю
- Если нет — создаем нового клиента

**Шаг 4. Нужно понять, что сказал клиент**
PHP говорит Python-сервису (по gRPC): "Распознай речь, вот аудио"
Python возвращает текст: "Алло, у меня просрочка по кредиту"

**Шаг 5. Нужно понять смысл**
PHP другому Python-сервису: "Что клиент хочет?"
Python: "Интент — проблема с платежом, срочность — высокая"

**Шаг 6. Принять решение**
PHP смотрит в сценарий:
- Если срочно и проблема с деньгами → перевести на оператора
- Если просто вопрос → ответить автоматически

**Шаг 7. Выполнить решение**
PHP либо:
- Говорит телефонии: "Переведи на оператора Иванова"
- Или просит третий Python-сервис: "Синтезируй речь для ответа"

**Шаг 8. Записать всё в логи**
PHP отправляет лог в OpenSearch: "Звонок 123 обработан за 2.3 секунды, решение: перевод"

**Шаг 9. Отправить метрику**
PHP шепчет Datadog: "Еще один звонок, длительность 2.3с, статус OK"

## Что упрощает этот проект?

1. **Для бизнеса**
   - Не нужно нанимать 100 операторов
   - AI работает 24/7 без обеда и больничных
   - Можно масштабироваться без найма людей

2. **Для разработчиков**
   - Четкое разделение: PHP координирует, Python думает
   - Каждый сервис можно менять независимо
   - Легко добавлять новые AI-модели

3. **Для инфраструктуры**
   - Горизонтальное масштабирование (просто добавить еще сервер)
   - Отказоустойчивость (упал Python — PHP переключит на запасной)
   - Мониторинг всего и вся

## Главная ценность проекта

PathVoice — это **мозг**, который соединяет:
- Голос клиента
- Умные AI-сервисы
- Бизнес-логику компании
- Операторов колл-центра

Он не делает всю работу сам, но **заставляет всех работать слаженно**.
