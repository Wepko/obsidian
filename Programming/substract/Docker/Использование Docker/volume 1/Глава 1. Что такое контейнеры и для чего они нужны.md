### **Глава 1. Что такое контейнеры и для чего они нужны**

#### **1. Сравнение контейнеров с виртуальными машинами**
**Вопросы для самопроверки:**
*   В чем заключается ключевое архитектурное различие между виртуальной машиной (ВМ) и контейнером?
*   Какие компоненты виртуальной машины (гипервизор, гостевые ОС) отсутствуют в модели контейнеризации?
*   Как различие в архитектуре влияет на скорость запуска, потребление ресурсов (память, диск) и производительность контейнеров по сравнению с ВМ?
*   Что обеспечивает изоляцию процессов в контейнере, если он использует ядро хоста?

**Что я изучил:**
*   **Виртуальная машина** виртуализирует *аппаратное обеспечение* (через гипервизор), поверх которого запускается полноценная гостевя ОС со своим ядром. Это дает сильную изоляцию, но требует много ресурсов.
*   **Контейнер** виртуализирует *операционную систему*. Все контейнеры на хосте используют общее ядро ОС хоста, но изолированы на уровне процессов, файловой системы и сети. Это делает их невероятно легковесными и быстрыми.
*   Контейнеры обеспечивают изоляцию с помощью механизмов ядра Linux: **namespaces** (изоляция) и **cgroups** (ограничение ресурсов).

**Краткие вопросы:**
*   Почему контейнер запускается за секунды, а виртуальная машина — за минуты?

#### **2. Docker и контейнеры**
**Вопросы для самопроверки:**
*   Является ли Docker синонимом понятия "контейнер"? Если нет, то в чем разница?
*   Какие три ключевые проблемы решает Docker в процессе разработки и эксплуатации ПО ("dependency hell", "works on my machine", environment consistency)?
*   Что такое **образ Docker** и чем он отличается от **контейнера**?
*   Какую основную философию или единицу упаковки предлагает Docker ("Build, Ship, Run")?

**Что я изучил:**
*   **Docker** — это платформа для разработки, доставки и запуска приложений в **контейнерах**. Это не единственная технология контейнеризации, но самая популярная.
*   Docker решает проблему несоответствия сред: приложение, его зависимости и конфигурация упаковываются в стандартный, переносимый образ.
*   **Образ (Image)** — это неизменяемый шаблон, "чертеж" с инструкциями для создания контейнера. **Контейнер (Container)** — это запущенный, изолированный экземпляр образа.
*   Логика: **Собрать (Build)** образ -> **Перевезти (Ship)** его в реестр -> **Запустить (Run)** где угодно.

**Краткие вопросы:**
*   Исправит ли Docker проблему, когда приложение работает на компьютере разработчика, но не работает на сервере? Почему?

#### **3. Краткая история Docker**
**Вопросы для самопроверки:**
*   Какие технологии и проекты предшествовали появлению Docker (например, chroot, LXC, Solaris Zones)?
*   Какое ключевое событие произошло в 2013 году, сделавшее Docker популярным?
*   Как изначальное название проекта (`dotCloud`) связано с его текущим именем?
*   Какая организация сейчас поддерживает и развивает Docker? Что такое Open Container Initiative (OCI)?

**Что я изучил:**
*   Идеи изоляции существуют давно (`chroot`, 1979). Непосредственным предшественником был **LXC** (Linux Containers).
*   Docker, основанный как внутренний проект компании `dotCloud` в 2013 году, представил простой инструментарий и форматы (Dockerfile, образ), которые сделали контейнеры доступными для масс.
*   Рост экосистемы привел к созданию **Open Container Initiative (OCI)** под эгидой Linux Foundation для стандартизации форматов образов и рантайма, что отделило спецификации от конкретной реализации Docker.

**Краткие вопросы:**
*   Какая организация стандартизирует форматы контейнеров сегодня?

#### **4. Дополнительные модули и надстройки**
**Вопросы для самопроверки:**
*   Что такое Docker Engine? Из каких основных компонентов он состоит?
*   Какую роль играют проекты **containerd** и **runc** в современной архитектуре Docker?
*   Что такое **Docker Desktop** и для кого он предназначен?
*   Какие дополнительные коммерческие и платные продукты предлагает Docker Inc. (Docker Hub, Docker Trusted Registry, теперь Docker Scout)?

**Что я изучил:**
*   **Docker Engine** — это сердце платформы (демон `dockerd`, API, CLI). В 2017 году его архитектура была модуляризована.
*   **containerd** — это выделенный высокоуровневый рантайм, который управляет жизненным циклом контейнеров (запуск, остановка, извлечение образов).
*   **runc** — это низкоуровневый инструмент, который непосредственно создает и запускает контейнеры в соответствии со спецификацией OCI.
*   **Docker Desktop** — это удобное приложение для macOS и Windows, которое включает в себя Engine, CLI, Compose и Kubernetes.

**Краткие вопросы:**
*   Какой компонент отвечает за непосредственный запуск контейнера согласно стандарту OCI?

#### **5. 64-битовая версия ОС Linux**
**Вопросы для самопроверки:**
*   Почему для работы Docker (на хост-машине) требуется 64-битная операционная система?
*   Какие основные требования к ядру Linux предъявляет Docker?
*   Может ли Docker работать на 32-битных системах или на других архитектурах процессоров (ARM, PowerPC)?
*   Почему Docker изначально был технологией для Linux, и как он работает на macOS и Windows?

**Что я изучил:**
*   Docker требует современного 64-битного ядра Linux (минимально 3.10+), так как использует его низкоуровневые функции (namespaces, cgroups).
*   На **macOS и Windows** Docker работает через легковесную виртуальную машину (гипервизор), внутри которой запускается полноценный Linux, и уже в нем — Docker Engine. Поэтому требование к 64-битной системе остается.
*   Docker поддерживает множество архитектур (x86-64, ARM, IBM Z), но образ должен быть собран под конкретную архитектуру.

**Краткие вопросы:**
*   Могу ли я запустить контейнер с образом, собранным для ARM, на ноутбуке с процессором Intel? Что для этого нужно?

#### **6. Резюме (Общее для главы)**
*   Контейнеры — это легковесная, быстрая и эффективная альтернатива виртуальным машинам, обеспечивающая изоляцию на уровне ОС.
*   Docker — это платформа, которая упростила использование контейнеров, решив классические проблемы переносимости и согласованности сред.
*   Современный Docker — это модульная экосистема с открытыми стандартами (OCI), состоящая из Engine, containerd, runc и множества вспомогательных инструментов.
*   Для работы Docker необходим современный 64-битный Linux или его виртуальная версия внутри macOS/Windows.
