### Ответы на вопросы для самопроверки:

**1. Что такое Docker Engine? Из каких основных компонентов он состоит?**

**Docker Engine** — это сердцевина Docker, опенсорс-движок, который создает и запускает контейнеры. Это клиент-серверное приложение, состоящее из трех основных компонентов:

1.  **Docker Daemon (`dockerd`)**: Серверный процесс (демон), который работает на хосте и управляет образами, контейнерами, сетями и томами. Он прослушивает запросы от клиента.
2.  **REST API**: Интерфейс, который программы могут использовать для взаимодействия с демоном и давать ему инструкции.
3.  **Docker CLI (Command Line Interface)**: Клиентская утилита командной строки, которую используют пользователи. Она отправляет команды (например, `docker run`) через REST API демону, который их выполняет.

**2. Какую роль играют проекты containerd и runc в современной архитектуре Docker?**

После модуляризации в 2017 году архитектура стала выглядеть как "стек", где каждый уровень отвечает за свою задачу:

*   **`runc` (нижний уровень)**: Это **низкоуровневый OCI-совместимый рантайм**. Его единственная задача — **непосредственно создать и запустить контейнер** из спецификации OCI-образа. Он является оберткой вокруг системных вызовов Linux для создания пространств имен и контрольных групп. `runc` — это эталонная реализация спецификации OCI Runtime. Он ничего не знает об образах или сетях, он просто запускает процессы.

*   **`containerd` (средний уровень)**: Это **высокоуровневый рантайм**. Он управляет полным жизненным циклом контейнеров на хосте. Его обязанности:
    *   Загрузка и управление OCI-образами (из реестра).
    *   Управление хранилищем.
    *   Управление выполнением контейнеров через `runc`.
    *   Оркестрация низкоуровневых операций (создание, запуск, остановка, удаление контейнеров).
    *   Предоставление стабильного API для верхнеуровневых систем (таких как Docker Engine или Kubernetes).

**Взаимодействие в Docker**: Когда вы даете команду `docker run`, запрос идет через Docker CLI -> Docker Daemon -> **containerd** -> **runc**. `containerd` выступает как менеджер и контроллер, а `runc` — как простой исполнитель.

**3. Что такое Docker Desktop и для кого он предназначен?**

**Docker Desktop** — это настольное приложение для **macOS и Windows**, которое упрощает установку и работу с Docker на компьютерах разработчиков.

Его ключевые особенности и целевая аудитория:
*   **Для кого:** В первую очередь, для **разработчиков и DevOps-инженеров**, работающих на macOS или Windows.
*   **Что включает:** Единый установщик, который содержит:
    *   Docker Engine (как виртуальную машину Linux на macOS/Windows).
    *   Docker CLI.
    *   Docker Compose.
    *   Встроенный Kubernetes-кластер.
    *   Графический пользовательский интерфейс (GUI) для мониторинга и управления.
    *   Интеграцию с системой.
*   **Главная задача:** Создать на локальной машине разработчика идентичную Linux-среду, где можно легко собирать, тестировать и отлаживать контейнеризированные приложения перед отправкой их на Linux-сервера.

**4. Какие дополнительные коммерческие и платные продукты предлагает Docker Inc. (Docker Hub, Docker Trusted Registry, теперь Docker Scout)?**

1.  **Docker Hub (SaaS, есть бесплатный и платные тарифы):** Крупнейший в мире облачный реестр (registry) для публичных и приватных образов Docker. Платные планы (Pro, Team) дают больше приватных репозиторий, повышенные лимиты на извлечение образов, автоматические сборки, инструменты для команд.
2.  **Docker Trusted Registry (DTR):** **Устаревший продукт.** Ранее это было локальное (on-premise) решение для хранения и управления образами внутри корпоративной сети. Его функциональность теперь интегрирована в **Mirantis Container Runtime (MCR)** и другие продукты Mirantis после приобретения.
3.  **Docker Scout (ключевой современный продукт):** Сервис **анализа безопасности образов** и управления их жизненным циклом. Он автоматически сканирует образы на наличие уязвимостей (CVE) в зависимостях, оценивает их, дает рекомендации по обновлению и показывает цепочку поставок (SBOM). Имеет как бесплатные, так и платные корпоративные функции.

### Ответ на краткий вопрос:
**Какой компонент отвечает за непосредственный запуск контейнера согласно стандарту OCI?**

За непосредственный запуск контейнера согласно стандарту OCI отвечает **`runc`**. Это минималистичный инструмент, который получает на вход JSON-конфигурацию (bundle) от вышестоящего менеджера (например, `containerd`) и с помощью системных вызовов создает изолированное окружение (namespaces, cgroups) и запускает в нем процесс.
