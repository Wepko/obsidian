Отличный вопрос! Ты хочешь понять, какие **стандарты** существуют в этой новой индустрии. Сейчас 2026 год, и индустрия уже прошла путь от полного хаоса ("каждый пишет промпты как хочет") до формирования вполне конкретных стандартов.

Вот основные стандарты, которые сложились в workflow с ИИ (и которые, кстати, использует AI Factory):

---

## 1. **Agent Skills Specification (agentskills.io)** — Формат навыков

Это **самый важный стандарт**, который определяет, как должны выглядеть инструкции для ИИ-агентов.

### Что это?
Спецификация формата `SKILL.md` файлов, которые агенты читают как "команды".

### Стандартная структура:

```markdown
---
name: skill-name           # Уникальное имя команды
description: Что делает     # Для поиска /help
argument-hint: "[options]" # Подсказка при табуляции
allowed-tools: Read Write  # Какие инструменты разрешены
disable-model-invocation: true/false  # Может ли агент вызывать сам
---

# Название навыка

[Инструкции...]
```

### Кто поддерживает:
- Claude Code
- Cursor
- Windsurf
- Roo Code
- Kilo Code
- И ещё ~10 агентов

### Зачем стандартизировали?
Раньше каждый агент имел свой формат:
- Claude Code: `.claude/commands/` с YAML frontmatter
- Cursor: `.cursor/rules/` с произвольным Markdown
- GitHub Copilot: `.github/copilot-instructions.md`

Теперь все переходят на **единый формат** с frontmatter.

## 2. **MCP (Model Context Protocol)** — Стандарт интеграции с инструментами

**Создан Anthropic**, но стал индустриальным стандартом.

### Что это?
Протокол, как ИИ-агенты общаются с внешними инструментами:
- Файловая система
- Базы данных (Postgres, MySQL)
- GitHub API
- Браузер (Chrome DevTools)
- И т.д.

### Стандартный формат конфига:

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" }
    }
  }
}
```

### Кто поддерживает:
Практически **все современные агенты**: Claude Code, Cursor, Windsurf, Roo Code, Cline, Continue, и даже VS Code через расширения.

### Почему это стандарт?
Раньше каждый агент изобретал свой API для тулов. MCP стал общим языком — один раз написал сервер, работает везде.

## 3. **Spec-Driven Development** — Стандарт процесса

Не формальный стандарт (как RFC), но **индустриальный паттерн**, который все копируют.

### Что это?
Идея: **сначала спецификация, потом код**.

### Стандартный цикл:
```
1. Исследование (Research) → RESEARCH.md
2. Планирование (Plan) → PLAN.md
3. Утверждение (Review) → человек проверяет план
4. Реализация (Implement) → код по плану
5. Верификация (Verify) → проверка против плана
6. Коммит (Commit) → структурированное сообщение
```

### Почему стал стандартом?
Опыт показал: если дать агенту задачу без плана, он:
- Либо делает слишком мало
- Либо уходит в бесконечный рефакторинг
- Либо ломает существующий код

План — это **ограничитель** и **чекпоинт**.

---

## 4. **Conventional Commits для ИИ** — Стандарт коммитов

### Что это?
Адаптация [Conventional Commits](https://www.conventionalcommits.org/) для ИИ-сгенерированных изменений.

### Стандартный формат:
```
feat: add user authentication
fix: resolve null reference in UserProfile
chore: upgrade dependencies
docs: update README with setup instructions
refactor: extract video processing service
perf: optimize whisper model loading
test: add integration tests for upload endpoint
```

### Почему важен?
- ИИ генерирует много коммитов
- Нужно быстро понимать, что изменилось
- Автоматическая генерация changelog'ов
- Интеграция с CI/CD (авто-релизы)

## 5. **Project Context Files** — Де-факто стандарт структуры

В индустрии сложились **общепринятые имена файлов** для контекста проекта:

| Файл | Назначение |
|------|------------|
| `DESCRIPTION.md` | Что это за проект, его цель |
| `ARCHITECTURE.md` | Как проект устроен технически |
| `RULES.md` | Правила и конвенции |
| `ROADMAP.md` | Планы развития |
| `RESEARCH.md` | Исследования, альтернативы |
| `PLAN.md` | Текущий план работ |
| `FIX_PLAN.md` | План исправления бага |
| `LESSONS.md` / `LEARNINGS.md` | Извлечённые уроки |

### Где используется:
- Все крупные проекты на GitHub
- Шаблоны для ИИ-разработки
- AI Factory, Cursor Rules, Windsurf Rules

## 6. **Итеративный цикл с Quality Gates** — Стандарт качества

### Что это?
Структурированный цикл генерации, где каждая итерация проверяется.

### Стандартные фазы:
```
PLAN → PRODUCE → EVALUATE → CRITIQUE → REFINE → (повтор)
```

### Quality Gates:
- **Синтаксис** (линтеры, компиляция)
- **Тесты** (юнит, интеграционные)
- **Соответствие спецификации** (проверка по плану)
- **Безопасность** (сканирование уязвимостей)
- **Производительность** (бенчмарки)

### Почему стандарт?
Раньше: "сгенерировал и надеюсь, что работает".
Теперь: "сгенерировал и доказал, что работает".

---

## 7. **Prompt Injection Defense** — Стандарт безопасности

### Что это?
Методология защиты от промпт-инъекций во внешних навыках.

### Двухуровневая защита (стандарт):

**Уровень 1: Статический анализатор**
- Ищет паттерны: `ignore previous instructions`, `system prompt`, `secretly`
- Проверяет base64, hex, zero-width символы
- Детектирует попытки кражи credentials

**Уровень 2: Семантический анализ (LLM)**
- Понимает контекст
- Ловит переформулированные атаки
- Оценивает "подозрительность" намерений

### Кто использует:
- Все маркетплейсы навыков (skills.sh, agent directories)
- Корпоративные политики безопасности
- Open-source сканеры

## 8. **Git Worktree для параллельных фич** — Стандарт организации

### Что это?
Использование `git worktree` для изоляции фич, над которыми работает ИИ.

### Стандартная команда:
```bash
git worktree add ../project-feature-auth feature/auth
```

### Преимущества:
- Полная изоляция кода
- Можно запускать отдельные инстансы
- Нет конфликтов при параллельной работе
- Легко удалить, если фича не взлетела

---

## 9. **SemVer для ИИ-навыков** — Стандарт версионирования

### Что это?
Адаптация Semantic Versioning для навыков ИИ.

### Формат:
```
aif-plan@2.1.0
```

**Правила:**
- **MAJOR** — ломающие изменения в интерфейсе навыка
- **MINOR** — новая функциональность, обратно совместимо
- **PATCH** — исправления, улучшения инструкций

### Где используется:
- Маркетплейсы навыков
- Автоматические обновления (ai-factory update)
- Зависимости между навыками

---

## 10. **Evolution Loop** — Стандарт самосовершенствования

### Что это?
Механизм, где система учится на своих ошибках.

### Стандартный цикл:
```
1. Баг → Фикс → Патч-файл с root cause
2. Накопление патчей
3. Анализ патчей → выявление паттернов
4. Обновление правил в навыках
5. Следующие запуски умнее
```

### Формат патча:
```markdown
# [Краткое описание]

**Date:** 2026-03-13
**Files:** [список файлов]
**Severity:** [low|medium|high]

## Problem
[Что пошло не так]

## Root Cause
[Почему это произошло]

## Solution
[Как исправили]

## Prevention
[Что добавить в навыки, чтобы не повторялось]

## Tags
`#null-check` `#api-design`
```

## Что это значит для тебя?

Когда ты создаёшь свою "ручную AI Factory" для Video2Text, ты можешь **сознательно следовать этим стандартам**:

1. Использовать **frontmatter** в своих инструкционных файлах
2. Хранить контекст в **стандартных именах файлов** (`DESCRIPTION.md`, `ARCHITECTURE.md`)
3. Писать коммиты в формате **Conventional Commits**
4. Использовать **итеративный цикл** с проверкой качества
5. Создавать **патчи** с уроками и учиться на ошибках

Это не просто "как я придумал", а **индустриальные практики**, которые доказали свою эффективность тысячи раз.
